# moodie-foodie

Moodie-Foodie is a mood-based food recommendation app built for urban Pakistan. It combines a FastAPI backend, MongoDB persistence, and a React + Vite frontend to turn moods into personalized restaurant dish suggestions.

## 🚀 What it does

- Converts an initial mood into follow-up questions using an LLM
- Refines mood and extracts dining tags like `comfort`, `spicy`, `healthy`, `cozy`
- Matches tags against a restaurant dish dataset
- Returns recommendations with reasoning and session tracking
- Supports login/signup and optional dietary preferences for authenticated users
- Records thumbs-up / thumbs-down feedback for recommendation sessions

## 🏗️ Architecture

- `backend/`
  - `main.py`: FastAPI app entrypoint, CORS, lifecycle startup/shutdown
  - `api/router.py`: central router and health endpoint
  - `api/routes/auth.py`: signup/login endpoints with JWT
  - `api/routes/recommendations.py`: generate questions and get recommendations
  - `api/routes/feedback.py`: submit feedback for an existing session
  - `database/mongodb.py`: MongoDB connection and index creation
  - `core/config.py`: environment settings and allowed vocabularies
  - `services/llm_service.py`: Groq calls and prompt handling
  - `services/matching_service.py`: tag matching against dish data
  - `services/recommendation_service.py`: recommendation pipeline orchestration
- `frontend/`
  - Vite + React app
  - `src/api/client.js`: centralized API wrapper with auth token support
  - `src/api/auth.js`: login/signup requests
  - `src/api/recommendations.js`: recommendation flow helpers

## ⚙️ Prerequisites

- Python 3.11+
- Node 18+ / npm
- MongoDB accessible via `MONGODB_URL`
- `GROQ_API_KEY` for Groq LLM calls

## 📦 Backend setup

1. `cd backend`
2. `python -m venv .venv`
3. `.venv\Scripts\activate` (Windows) or `source .venv/bin/activate` (macOS/Linux)
4. `pip install -r requirements.txt`
5. Create a `.env` file in `backend/` with at least:

   ```text
   MONGODB_URL=mongodb://admin:secret@localhost:27017/foodie_moodie?authSource=admin
   MONGODB_DB_NAME=foodie_moodie
   GROQ_API_KEY=your_groq_key_here
   SECRET_KEY=replace_this_in_production
   ```

6. Verify `backend/data/restaurants.json` exists and contains the menu dataset.

## ▶️ Run backend

From `backend/`:

```bash
uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

Or you can simply run:
```bash 
python main.py
```

Then visit `http://127.0.0.1:8000/docs` for automatic OpenAPI docs.

## 🌐 Frontend setup

1. `cd frontend`
2. `npm install`
3. `npm run dev`
4. Open `http://localhost:5173`

If the backend runs on a different address, set `VITE_API_URL` in `frontend/.env`:

```text
VITE_API_URL=http://localhost:8000/api
```

## 🔐 Authentication

The backend supports signup and login via JWT.

- `POST /api/auth/signup`
- `POST /api/auth/login`

The frontend stores the token in `localStorage` and sends it in `Authorization: Bearer ...` headers automatically for authenticated requests.

## 📡 API Endpoints

- `GET /api/health`
  - health check for the API
- `POST /api/recommendations/generate-questions`
  - request body: `{ "mood": "stressed" }`
  - returns follow-up questions generated by the LLM
- `POST /api/recommendations/get-recommendations`
  - request body includes `initial_mood`, `answers`, and optional `dietary_override`
  - returns `session_id`, `refined_mood`, `refined_tags`, and recommendations
- `POST /api/feedback/submit`
  - request body: `{ "session_id": "...", "thumbs_up": true }`
  - records feedback in the matching session document

## 🧩 Data persistence

The application persists data in MongoDB:

- `users`
- `user_preferences`
- `sessions`

Indexes are created automatically at startup.

## 🔧 Notes

- `core/config.py` defines valid moods, valid tags, dietary flags, and MongoDB defaults
- `database/mongodb.py` initializes the Motor client and ensures collection indexes
- `frontend/src/api/client.js` centralizes fetch calls and auth header injection

## ✅ Quick sanity checks

- `GET /api/health` returns a successful response
- `POST /api/recommendations/generate-questions` returns questions
- `POST /api/recommendations/get-recommendations` returns recommendation payload
- `POST /api/feedback/submit` records session feedback
